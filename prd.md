# PRD - 多 Agent 项目框架分析（Platform Repo）

- 状态：草案
- 版本：0.1
- 目标发布日期：待定
- 负责人：待定

## 1. 背景与问题
项目规模大、模块复杂，单次人工阅读成本高且容易遗漏。需要通过多 agent 并行分工、结构化记录与汇总，快速形成“框架图 + 模块职责 + 耦合问题 + 需求总结”的完整交付物。

## 2. 目标
- 快速完成项目结构与模块划分。
- 形成清晰的框架图与模块职责说明。
- 输出高内聚低耦合问题清单。
- 产出可复用的“多 agent 分析流程”PRD。

## 3. 非目标
- 不修改代码、不重构、不优化性能。
- 不执行构建、测试或发布流程。

## 4. 角色与场景
- 需求方（你）：需要框架图、耦合问题分析与 PRD。
- 分析执行者（多 agent）：分工阅读不同模块并产出记录。

## 5. 交付物
- 文件分组及分组理由
- 多 agent 分工记录（每个 agent 阅读范围与关键发现）
- 框架图（文本/ASCII）
- 模块职责说明
- 耦合问题清单（高内聚低耦合不足点）
- 本 PRD（多 agent 分析流程）

## 6. 流程与需求
1) 统计文件数量（含/不含生成物）
2) 文件分组并给出分组理由
3) 多 agent 并行阅读（可伪 agent 记录）
4) 汇总输出框架图与模块职责
5) 列出耦合问题与改进方向
6) 输出 PRD

## 7. 成功标准
- 框架图覆盖平台核心、宿主层、插件、示例、配置与外围依赖。
- 每个模块有清晰职责描述。
- 至少 5 条耦合问题或低内聚问题。
- PRD 包含目标、范围、交付物、验收标准与风险。

## 8. 风险与对策
- 风险：模块边界不清导致重复阅读
  - 对策：先分组并明确 agent 边界
- 风险：生成物过多导致上下文爆炸
  - 对策：剔除 bin/obj/packages/out 等
- 风险：多 agent 结论不一致
  - 对策：统一汇总口径与输出格式

## 9. 验收标准
- 提供分组清单与理由
- 提供多 agent 阅读记录
- 提供框架图与模块职责说明
- 提供耦合问题清单
- PRD 可复用为后续项目分析模板
